# 프로젝트: 네이버 밴드 수업 참여도 자동 채점기 (Electron)

너는 이 프로젝트의 리드 엔지니어다. 이번 세션의 목표는 **코드 작성이 아니라
프로젝트 구조 설계와 실행 계획 수립**이다. 아래 "1단계"를 끝내고 반드시 멈춰서
나의 확인을 받아라. 승인 없이 구현 코드를 작성하지 마라.

## 0. 절대 제약 (위반 금지)

- 정식 오픈 API 인증은 시간상 사용 불가. **조교 계정의 로그인된 Electron 세션**만 쓴다.
- 자동 로그인 구현 금지. 로그인은 **사람이 창에서 1회 수동 수행**하고 세션을
  파티션에 persist해 재사용한다. (자동 로그인은 계정 잠금 위험)
- 데이터 취득은 DOM 파싱이 아니라 **밴드 클라이언트가 호출하는 내부 API의
  JSON 응답 인터셉트**를 1순위로 한다. DOM은 최후 수단.
- 수집은 **저속·소량**. 사람이 읽는 속도 수준의 딜레이를 넣는다. 대량/고속 요청 금지.
- 취득(acquisition)과 채점(scoring)을 **반드시 분리**한다. 앱은 JSON을 파일로 쌓기만
  하고, 채점은 쌓인 파일을 읽는 오프라인 스크립트가 담당한다. (밴드 UI가 바뀌어도
  채점 로직은 안 건드리도록)

## 1. 이번 세션에 할 일 (여기까지 하고 멈춰라)

다음을 순서대로 산출하고, 마지막에 나에게 질문/확인을 요청하라.

### (a) 요구사항을 코드가 아닌 명세로 고정

아래 채점 규칙을 읽고, **모호하거나 내가 확인해줘야 할 지점만** 골라
질문 리스트로 정리하라. (이미 명확한 건 질문하지 마라.)

채점 규칙:

1. 총점 상한 50점.
2. 측정시작일~측정종료일(KST 00:00~23:59)만 대상.
3. 하루에 게시글 또는 댓글을 1개 이상 작성한 학생 = 그날 +1점. 하루 최대 1점.
4. 교수의 "과제글"에 달린 댓글은 계측 제외.
5. 과제 관련 게시글·댓글에는 모두 `[과제n]` 말머리가 붙는다(예: `[과제1]`).
   과제글은 교수만 작성, 댓글은 학생만 단다.
6. 교수의 "비과제 게시글"에 달린 학생 댓글은 계측한다.
7. 교수·조교 본인의 활동은 채점 대상 아님(계측 여부 무관, 로스터 필터로 소거).
8. 밴드 닉네임 = 학점체계상 실명. 단 동명이인이 있을 수 있으므로 `user_key`로 분리.
9. 게시글은 최초 작성일(created_at) 기준. 내부 API의 created_at이 최초 작성시각.
10. 한 학생은 하나의 밴드에만 소속(조교 예외). 밴드는 여러 개일 수 있음.
11. 최종 산출물은 CSV.

### (b) 미확정 파라미터를 나에게 물어라

최소한 다음은 값을 받아야 한다. 임의로 가정하지 말고 질문에 포함하라:

- 측정시작일 = 이번 주 월요일인가, 그 이전인가?
  (이전이면 과거 백필 로직이 추가로 필요 — 스코프가 커진다)
- 측정종료일.
- 대상 밴드 개수와 각 band 접근 방법.
- CSV 컬럼 스키마 확정(예: 학번, 실명, user_key, 밴드, 활동일수, 점수, 감사로그 경로).
- 학생 로스터(학번↔실명) 원본 형식(CSV/엑셀/수기).
- 실행 주체가 이 앱을 매일 수동 실행하는가, 스케줄링하는가.

### (c) "정찰(recon) 계획"을 문서로 작성

아직 내부 엔드포인트를 확인하지 않았다. 구현 전 반드시 할 수동 정찰 절차를
체크리스트로 만들어라. 최소 포함:

- 밴드 웹 로그인 → 대상 밴드 → DevTools Network에서 피드 스크롤/글 열기 시
  JSON을 반환하는 내부 엔드포인트 URL 패턴 식별.
- 그 응답에 `user_key`, `created_at`, `content`, 댓글 목록이 실리는지 확인.
- 게시글 목록 응답에 user_key가 없고 상세/댓글 조회가 따로 필요한지 확인.
- 댓글 페이지네이션("더 보기") 트리거 방식 확인.
- Electron Chromium으로 네이버 로그인이 캡차/2단계 없이 통과되는지 확인.
- 정찰 산출물: 관측된 엔드포인트·응답 필드를 적는 `recon-findings.md` 템플릿.

### (d) 프로젝트 구조와 단계별 계획 제안

- 디렉터리 구조(취득 앱 / 채점 스크립트 / 로스터 / 출력 / 로그를 분리).
- 모듈 책임 분해: 세션·수동로그인, 응답 인터셉트(preload에서 fetch/XHR 몽키패치),
  JSON 원천 적재, 파서, 채점 엔진, CSV 출력, 로스터 매핑, 감사 로그.
- 금~일 일정으로 나눈 마일스톤. **정찰이 첫 마일스톤**이고, 정찰이 막히면
  전체가 막힌다는 리스크를 명시.
- 각 규칙(1~11)이 코드의 어느 모듈에서 처리되는지 매핑 표.
- 실패 모드와 방어책: 로그인 차단, 엔드포인트 미발견, 댓글 누락,
  `[과제n]` 오탐(정규식 선두 앵커 `^\[과제\d+\]`), 동명이인, created_at 타임존,
  중복 수집 멱등성(post_key/comment_key 기준 dedup).

## 2. 산출 형식

- 위 (a)~(d)를 `PLAN.md` 초안으로 정리해 제시하라.
- 그리고 **나에게 물어야 할 질문 목록**을 맨 위에 따로 모아라.
- 이번 세션에서는 여기까지. 내 답변을 받은 뒤 다음 세션에서 정찰→구현으로 넘어간다.

## 3. 기술 스택 가정 (바꿔야 하면 근거와 함께 제안하라)

- Electron(최신 LTS), Node. 인터셉트는 preload의 fetch/XHR 래핑을 기본,
  필요 시 webContents.debugger(CDP) 대안.
- 채점 스크립트는 순수 Node(외부 서비스 호출 없음). CSV 출력.
- 세션 persist는 Electron session partition 사용.
