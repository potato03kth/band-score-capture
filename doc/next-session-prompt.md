## 이번 세션 목표 — M4: 다중 밴드 확장 + 실사용 준비 (`doc/PLAN.md` D-4 마일스톤 표)

> 게이트: **로스터 전원 매핑상태 확인.**

### 우선순위 순서 (제안 — 사용자와 먼저 상의)

1. **실제 로스터 학번 입력.** 지금 `2_로스터.xlsx`에 학번이 없어 109명 전원이 `TEST####`
   임시 학번(`매핑상태:synthetic`)으로 나온다. 교수가 가진 실제 "수업참여 학생 리스트"(학번↔
   실명)를 받아 채워야 진짜 학번이 붙고, **이때 처음으로 동명이인(`ambiguous`) 판정 로직이
   실데이터로 검증된다** — 동명이인이 실제로 나오면 그 처리 워크플로(수기 확인)도 같이
   점검할 것.
2. **`수집완료여부` 오탐 조사.** 두 밴드 모두 `collection_status.json`의 `feedExhausted`가
   `false`로 남아있어 CSV에 "미완료(확인 필요)"로 표시된다(§I-1) — 실제로는 tier-2 피드
   스크롤 교차검증이 정확히 일치해 완주로 보이지만, `acquire/collector.js`의 종료 감지가
   최고령 글 근처에서 깔끔하게 "다음 글 버튼 없음"을 못 잡고 "게시글 상세 캡처 실패"로
   빠지는 것으로 보인다. **CLAUDE.md 원칙대로 추측성 수정 금지** — Electron을 띄워 두 밴드
   각각 최고령 글(3분반 최초 글, 4분반 최초 글)까지 실제로 스크롤해서 사용자가 직접 DOM/버튼
   상태를 봐야 원인이 갈린다. 이것만을 위해 별도 재기동할지, 다른 필요한 재기동에 묻어갈지
   사용자와 상의.
3. **다중 밴드 확장(2~7개).** 지금은 2개 밴드만 실사용 중 — 나머지 밴드가 있다면
   `input/1_설정.xlsx`의 "대상밴드" 시트에 추가하고 동일한 흐름(재수집→채점→게이트→수기
   대조)을 반복.
4. **디스퓨트 대응 리허설(U10).** 감사 시트가 실전에서 잘 작동함은 이미 확인됨(§I-3의 버그
   자체를 감사 시트를 사람이 읽다가 발견했다) — "학생이 이의제기하면 어떻게 근거를 보여주나"
   시나리오를 실제로 한 번 리허설.
5. **최종 CSV 확정.** 위 항목들이 끝나면 실제 배포용 최종 채점 결과를 산출.

### 참고

- `acquire/`는 CLAUDE.md 규칙대로 **여전히 손대지 않는 게 기본값**이다 — 위 2번처럼 실제
  acquire 버그가 의심되면 `doc/m1-live-findings.md`·`doc/m1-cdp-migration-plan.md` 맥락을
  먼저 읽고 신중하게 접근할 것.
- 실기동(Electron)이 필요하면 매번 **재수집 비용**(두 밴드 전체 재순회, 로그인/2FA 필요할 수
  있음)을 사용자에게 미리 알리고 진행 여부를 확인할 것 — M3 세션에서도 이 절차를 두 번
  거쳤고 매번 사용자 확인 후 진행했다.
- `score/`·`lib/` 쪽 로직 버그를 또 찾을 가능성보다는(M2에서 이미 136건 단위테스트 +
  M3 실데이터 관통으로 깊게 검증됨), **실데이터 특유의 이슈**(로스터 학번 매핑 실사례,
  동명이인 실사례, `acquire`의 종료 감지 엣지케이스)에 집중하는 게 맞을 가능성이 높다.
