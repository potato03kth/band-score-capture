### 확인·진행 순서 (제안 — 사용자와 먼저 상의)

1. **데이터가 최신인지 확인.** 마지막 날짜 폴더가 2026-07-20인데 오늘 날짜와 측정 종료일이
   그 이후라면, 재수집(백필 아님 - 증분)이 필요한지 사용자에게 먼저 물어라. **Electron을
   다시 띄우기 전에**, 이미 있는 트레이스/로그로 "언제까지 수집됐는지"·"수집이 완료 상태로
   끝났는지"(`collection_status.json`의 `feedExhausted`)부터 확인해 불필요한 재기동을 피해라.
2. **재수집이 필요하면**: `BSC_TRACE=1 npx electron .`. 수동 로그인이 필요할 수 있음을
   미리 사용자에게 안내(2FA 포함, 자동 스킵 보장 안 됨 - CLAUDE.md 명시). 재시작 최소화
   원칙을 지켜라(가설 하나당 재기동 금지 - 트레이스로 먼저 분석).
3. **`node score/index.js --mode=production` 실행.** `1_설정.xlsx`의 측정기간이 여전히
   유효한지부터 확인(스테일 설정으로 돌아가지 않도록). Phase 6 게이트(`부적합_데이터_확인.xlsx`)
   ·Phase 8(`score_logic.xlsx`)가 새로 부트스트랩되며 채점이 한 번 막힐 수 있다 — 정상
   동작이니 당황하지 말 것. `input/`의 노란칸을 실제로 확인·기입하는 과정 자체가 M3의
   일부다(교수 워크플로를 실제로 밟아보는 것).
4. **결과 검증 = CLAUDE.md 신뢰도 tier 순서를 지켜라.** `out/결과_및_감사_*.xlsx`의 점수를
   최종 확정하기 전에, tier-1(사람이 직접 밴드 화면을 보고 셈)로 소수 학생을 골라 수기 대조
   하는 게 D-4 게이트다. tier-3(밴드 API 필드)나 tier-4(수집 루프 종료 신호)만으로 "맞다"고
   판단하지 마라.
5. 불일치가 나오면 그 원인이 acquire(취득 누락)인지 score(채점 로직 버그)인지 먼저 구분하고,
   `doc/m1-live-findings.md`(acquire 쪽 이미 확인된 이슈들)부터 대조해 이미 알려진 이슈인지
   확인한 뒤 새 버그로 다룰지 판단해라.

### 참고

- `score/` 로직 자체(파서·규칙·채점기·로스터·gaps·score_logic)는 이번 세션 이전에 합성
  데이터로 이미 상당히 깊게 검증됨(단위테스트 136건 + E2E 스크래치 하네스 반복 실행) — M3에서
  또 같은 종류의 로직 버그를 찾기보다는, **실데이터 특유의 이슈**(엣지 케이스 데이터 형태,
  실제 동명이인, 실제 결손 패턴, 실제 excel 왕복 UX)에 집중하는 게 맞을 가능성이 높다.
- `acquire/`는 CLAUDE.md 규칙대로 **여전히 손대지 않는 게 기본값**이다 — M3에서 acquire
  버그가 실제로 발견되면 그때 가서 `doc/m1-live-findings.md`·`doc/m1-cdp-migration-plan.md`
  맥락을 먼저 읽고 신중하게 접근할 것(이 문서들이 왜 그렇게 조심스러운 태도를 요구하는지는
  CLAUDE.md의 "Working style" 절 참고).
